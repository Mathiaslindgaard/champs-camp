Prototype Approach

Which strategy for building the "show the club" version?

A

Next.js + hardcoded mock data

Full Next.js app with all 6 tabs, real design system, and hardcoded seed data (the members, sessions, events from the PRD). Deploy to Vercel — club tries it on their phones. When approved, swap mock data for real Supabase calls. Same codebase, no throwaway work.

Pros

  • Deployable to Vercel immediately
  • Real mobile experience
  • Zero throwaway code — database wiring is additive
  • Looks and feels like the real thing

Cons

  • More setup than plain HTML
  • Interactions are visual only (booking a session doesn't "save")
B

Next.js + localStorage state

Same as A but button presses actually persist locally. Book a session and the count goes up. Mark a shift as available and it appears in the list. Data lives in the browser — resets if you clear it.

Pros

  • More convincing interactions
  • Multiple people can "try" it independently

Cons

  • More implementation work now
  • localStorage logic gets thrown away when real DB lands
C

Vanilla HTML/CSS/JS

Static prototype, fastest to build. But no routing, no mobile-native feel, and you'd be throwing it all away when building the real app.

Pros

  • Fastest to produce

Cons

  • Doesn't feel like a real app on mobile
  • 100% throwaway code

My recommendation: Option A — builds the real foundation, ships to Vercel, and requires zero rework when the club says yes.
