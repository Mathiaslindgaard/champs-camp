Design: Architecture & Data Layer

How the prototype is structured — confirm before we write the plan

Tech Stack

Framework Next.js 14 (App Router) + TypeScript
Styling Tailwind CSS + CSS custom properties (brand colors)
Fonts Bebas Neue (headings/numbers) + DM Sans (body)
Data Hardcoded TypeScript mock data (lib/mock-data.ts)
State React useState — interactions work visually within a session
Deployment Vercel (push to GitHub → auto-deploy)

Mock Data Strategy

All seed data from the PRD lives in lib/mock-data.ts — members, sessions, events, shifts, handbook entries, chat messages. The app imports this directly, no database calls.

When the club approves and we add Supabase, we replace each mock import with a real supabase.from(...).select() call. The components themselves don't change.

Interaction Behaviour

UI state is managed with useState in each page component. This means:

  • ✅ Booking a session updates the count on screen
  • ✅ Taking a shift removes it from the available list
  • ✅ Secret handbook entries reveal/hide on tap
  • ✅ Workout generator produces a plan
  • ✅ Chat shows messages (pre-seeded, can type new ones in session)
  • ⚠️ Nothing persists on page refresh — fine for a demo

No Auth for Prototype

The prototype shows the Mathias L. (admin) view by default — all role-gated features are visible. A simple role switcher in the header lets you preview other roles (member, volunteer, trainer) to demo the access differences to the club.
